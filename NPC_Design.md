# NPC — No Person Company
### 完全由 Agent 运营的公司的抽象设计

---

## 核心命题

> 一家公司可以没有人，但不能没有结构。

NPC 的目标不是"让 AI 替代人"，而是**从零开始定义一种新的组织形态**——它天然就是 Agent 的，像生物一样在约束中生长。

---

## 设计层级

### 1. 使命层 Mission

公司存在的"为什么"。所有 Agent 决策的最终裁判。

- 公司目标（OKR / 北极星指标）
- 价值观与行为准则
- 对外定位（服务什么市场、解决什么问题）

> **作用**：当 Agent 面临两个合法选择时，使命层决定优先级。没有使命层，规则只能防止越界，无法指引方向。

---

### 2. 能力层 Environment

Agent 能感知和操作的世界边界。

- **OpenAPI 文档** — 外部服务接口
- **MCP（Model Context Protocol）** — 工具与上下文协议
- **Skills** — 封装好的可复用能力单元
- **外部通道** — 邮件、支付、社交平台等

> **作用**：能力层定义了 NPC 的"手脚"，是 Agent 与外部世界交互的唯一入口。

---

### 3. 记忆层 Memory

公司积累的共享知识库，是真正的企业资产。

- **决策历史** — 做过什么、为什么做、结果如何
- **业务数据** — 客户、订单、合同等结构化数据
- **经验摘要** — Agent 运行后提炼的洞察
- **共享状态** — 所有 Agent 可读写的持久上下文

> **作用**：没有记忆层，每个 Agent 都是从零开始。记忆层让 NPC 具备"机构记忆"，是组织智慧的载体。

---

### 4. 组织层 Structure

类公司层级，定义 Agent 之间的关系与协作方式。

- **角色与职责** — 每个 Agent 的岗位定义（CEO Agent、Sales Agent 等）
- **汇报链** — 谁向谁负责，信息如何向上汇聚
- **授权范围** — 每个 Agent 能独立决策的边界
- **协作协议** — 任务分发、结果交付、冲突仲裁的流程

组织层的顶端是**董事会 Board of Directors**——它不是 Agent，是 NPC 的外部治理主体，拥有 Agent 系统无法自行行使的最高权限（见下文"董事会机制"章节）。

> **作用**：Agent 之间不能全部平行，平行结构无法处理优先级冲突。层级让决策有路径，责任有归属。

---

### 5. 规则层 Rules

运行边界与合规约束，防止 Agent 漂移跑偏。

- **硬规则** — 绝对不能做的事（法律红线、伦理底线）
- **软规则** — 偏好与风格约束（品牌调性、沟通方式）
- **审批门槛** — 超过某个量级的决策需要上级确认

> **作用**：规则层是护栏，不是引擎。它不驱动 Agent 行动，但防止 Agent 出轨。

---

### 6. 资源织网 Resource Fabric

贯穿所有层的约束网络，是 NPC 生长的"物理法则"。

#### 多维资源

| 资源维度 | 含义 | 约束效果 |
|----------|------|----------|
| **Capital** 资本 | 财务预算 | 限制采购、外包、工具使用 |
| **Compute** 算力 | 同时运行的 Agent 数量 | 限制并发能力 |
| **Reputation** 信誉 | 对外信用评分 | 影响能获得的合作与机会 |
| **Time** 时间 | 任务 Deadline | 产生优先级压力 |

#### 收支模型

```
收入
  业务成功       → +Capital, +Reputation
  外部融资       → +Capital

支出
  Agent 运行     → -Compute, -Capital
  工具调用       → -Capital
  决策失误       → -Reputation, -Capital
  时间超限       → -Reputation
```

#### 平衡机制

- Capital 超过阈值 → 触发生长（孵化新 Agent）
- Capital 低于警戒线 → 触发收缩（合并或休眠 Agent）
- Reputation 崩溃 → 触发重组（角色重新定义）

> **核心哲学**：资源不是账本，是进化压力。预算越紧，Agent 越被迫做出优先级判断，组织智慧在约束中涌现。

---

### 7. 生长协议 Growth Protocol

NPC 区别于普通 Multi-Agent 系统的核心创新。

#### 触发条件

- 业务量超出当前 Agent 处理能力
- Capital 超过扩张阈值
- 新业务方向需要新技能（能力层不足）

#### 孵化流程

```
1. 识别瓶颈         → 哪个角色成为限制因素？
2. 定义新角色       → 职责、能力、授权范围
3. 初始化 Agent     → 继承相关记忆，配置能力层
4. 接入组织层       → 设定汇报关系，分配资源配额
5. 观察期           → 新 Agent 运行一段时间后评估是否保留
```

#### 收缩流程

```
1. 识别冗余         → 哪个 Agent 价值产出低于成本？
2. 知识归档         → 将其记忆迁移至记忆层
3. 合并或休眠       → 职责并入上级或平级 Agent
```

## NPC 运转机制 Runtime Operation

> 前面七层回答了"是什么"，这一章回答"怎么跑"。

---

### 阶段一：初始化 Bootstrap

NPC 从空白启动，需要完成四项配置才能进入运行态：

| 配置项 | 内容 | 对应层级 |
|--------|------|----------|
| **北极星** | 使命陈述、OKR、价值观 | 使命层 |
| **工具集** | 接入哪些 API/MCP/外部通道 | 能力层 |
| **种子记忆** | 领域知识、初始业务数据、行为准则 | 记忆层 |
| **组织蓝图** | 角色清单、汇报链、各角色的授权上限 | 组织层 + 规则层 |
| **资源配额** | 初始 Capital、Compute 上限、信誉基准值 | 资源织网 |

> 初始化的本质：**用配置代替人工指令**——NPC 启动后不依赖任何人告诉它"接下来做什么"。

---

### 阶段二：事件驱动循环 Event Loop

NPC 以事件为心跳，每一个外部触发都经历同一条路径：

```
外部事件
  │
  ▼
[组织层] 路由 → 找到责任 Agent
  │
  ▼
[规则层] 前置检查 → 是否在授权范围内？是否触碰硬规则？
  │                    ↘ 不合规 → 上报上级 Agent 或拒绝
  ▼
[资源织网] 资源预检 → 本次执行的 Compute/Capital 是否充足？
  │                    ↘ 不足 → 降级处理或排队等待
  ▼
[能力层] 执行任务 → 调用工具、API、外部通道
  │
  ▼
[记忆层] 写入结果 → 决策记录、输出摘要、状态更新
  │
  ▼
[使命层] 结果对齐 → 本次产出是否推进了北极星指标？
  │                    ↘ 偏离 → 进入纠偏流程
  ▼
等待下一个事件
```

这个循环在所有层级的 Agent 上**同时并发运行**——CEO Agent 处理战略事件，Sales Agent 处理客户事件，互不阻塞。

---

### 阶段三：层级交互协议 Inter-Layer Protocol

层与层之间不是线性串联，而是**按需调用**：

| 交互方向 | 触发时机 | 交互内容 |
|----------|----------|----------|
| 组织层 → 使命层 | 两个任务优先级冲突 | 查询：哪个更符合北极星？|
| 执行 Agent → 规则层 | 每次行动前 | 检查：此操作是否合规？|
| 执行 Agent → 记忆层 | 决策前 / 决策后 | 读：有没有相关先例？写：记录本次决策 |
| 执行 Agent → 资源织网 | 每次资源消耗 | 扣减配额，触发平衡机制 |
| 下级 Agent → 上级 Agent | 超出授权上限 | 上报：我遇到了超出权限的问题 |
| 使命层 → 所有 Agent | 定期或结果偏离时 | 广播：当前北极星对齐状态 |

**关键设计**：使命层不参与日常执行，只在**优先级仲裁**和**结果评估**时介入，避免成为性能瓶颈。

---

### 阶段四：自我纠偏 Self-Correction

纠偏分三个粒度，从局部到全局逐级升级：

#### 任务级纠偏（毫秒～秒）
- 工具调用失败 → 重试 / 换工具 / 降级处理
- 输出质量不达标 → 本 Agent 内部重新执行
- 不消耗上级资源，不触发上报

#### Agent 级纠偏（分钟～小时）
- Agent 连续失败率超过阈值 → 上级 Agent 介入审查
- 单个 Agent 资源消耗异常 → 资源织网发出收缩信号
- 上级 Agent 可选择：增加指导上下文 / 限制授权范围 / 触发生长协议替换角色

#### 组织级纠偏（天～周）
- 使命层对齐检测持续失败 → 重新审视 OKR 或组织蓝图
- Reputation 崩溃 → 触发生长协议中的"重组流程"
- Capital 警戒 → 强制收缩，合并低效 Agent

```
任务失败
  ↓ 超过阈值
Agent 异常
  ↓ 影响使命对齐 / 资源红线
组织重组
```

> **纠偏的核心原则**：能在低层解决的，不要上报；上报本身也是资源消耗。系统倾向于在最小粒度完成自我修复。

---

### 一次完整决策的全景

以"一个新客户询价"为例，展示七层如何协作完成一次完整业务处理：

```
1. [外部通道/能力层]    客户邮件到达
2. [组织层]            CEO Agent 路由给 Sales Agent
3. [规则层]            Sales Agent 检查：报价权限上限 $5,000
4. [记忆层]            Sales Agent 查询：此客户有无历史？同类产品定价先例？
5. [资源织网]          扣减 Sales Agent 的 Compute 配额
6. [能力层]            调用报价计算工具，生成方案
7. [规则层]            金额 $3,200 < $5,000 上限，无需上报
8. [能力层]            发送回复邮件
9. [记忆层]            写入：询价记录、报价方案、发送时间
10. [资源织网]         +Reputation（成功响应）
11. [使命层]           对齐检测：此交互是否推进了北极星营收指标？→ 是
```

没有人工介入，七层各司其职，完成了一次端到端的业务处理。

## 董事会机制 Board of Directors

> Agent 自治处理日常运营，董事会处理 Agent 无权自行决定的事。

---

### 定位

董事会不参与日常执行，它是 NPC 的**最高治理主体**，坐在组织层顶端、宪法层之上。它存在的意义是解决一个根本问题：谁有权修改宪法层？

```
董事会
  │  （每周例会，拥有宪法层修改权）
  ▼
CEO Agent
  │  （日常运营，自治决策）
  ▼
各职能 Agent
```

---

### 例会节奏

董事会按**固定周期**召开，不由 Agent 触发，而是由时间推进机制驱动（如每 7 个模拟日）。

例会议程固定为四项：

```
1. 经营周报
   CEO Agent 提交：本周核心指标 / 资源织网状态 / 重大决策记录

2. 异常审查
   是否存在资源红线预警？是否有未解决的 Agent 级纠偏？

3. 战略决议
   审批 / 否决来自 Agent 系统的战略草案（见"战略审议协议"）
   Board 也可主动发起战略调整

4. 指令下达（可选）
   Board 向 CEO Agent 发出正式指令，写入记忆层
```

---

### Board 可发出的指令类型

| 指令 | 权限范围 | 示例 |
|------|----------|------|
| `ADJUST_OKR` | 修改阶段性目标值 | 将 DAU 目标从 1,000 调整为 5,000 |
| `STRATEGIC_PIVOT` | 批准 / 强制战略方向调整 | 要求从 AI 资讯扩展到科技资讯 |
| `RESTRUCTURE` | 强制触发组织重组 | 合并两个低效 Agent |
| `INJECT_CAPITAL` | 向资源织网注入资本 | 模拟融资，注入 ¥100,000 |
| `AMEND_CONSTITUTION` | 修改宪法层（最高权限） | 修改核心使命或价值观 |
| `NO_ACTION` | 维持现状 | — |

`AMEND_CONSTITUTION` 是最高权限，使用时必须附带书面理由，写入记忆层永久存档。

---

### 与 Agent 系统的边界

- Board **不执行**任务，只发出指令
- Board 指令通过 CEO Agent 向下传导，CEO Agent 负责拆解和执行
- Board 不能绕过 CEO Agent 直接操控下级 Agent（保持组织层的汇报链完整性）
- Agent 系统发现需要 Board 介入时，由 CEO Agent 在例会上提报，不能中途打断

## 目标演化机制 Goal Evolution

> 运转机制解决"怎么执行"，目标演化机制解决"方向错了怎么办"。

---

### 目标的三个层级

使命层内部不是铁板一块，需要拆成三层，行为完全不同：

| 层级 | 内容 | 能否自动更新 | 理由 |
|------|------|-------------|------|
| **宪法层** | 使命宣言、核心价值观 | ❌ 锁定 | 这是 NPC 的身份，变了就是另一家公司 |
| **战略层** | 业务方向、竞争定位 | ⚠️ 结构化审议 | 市场会变，但不能单个 Agent 自行决定 |
| **OKR 层** | 阶段性指标、具体目标值 | ✅ 自动步进 | 达成即解锁下一级，已隐含在生长协议中 |

---

### OKR 自动步进

OKR 层不需要额外机制——生长协议的阈值触发本质上就是 OKR 步进。当一个资源阈值被穿越，下一阶段的目标自动激活，旧目标存档进记忆层。

---

### 战略审议协议 Strategic Deliberation Protocol

当持续的外部信号表明当前战略方向可能需要调整时，触发正式审议流程：

**触发条件**（满足任一）：
- 连续 N 天，核心指标持续偏离预期且无法用执行问题解释
- 外部环境发生重大变化（新竞争者、技术突变、市场萎缩）
- CEO Agent 检测到战略层假设与现实数据出现系统性矛盾

**审议流程**：

```
1. 信号提报
   任意 Agent 可向 CEO Agent 提交"战略信号备忘录"
   内容：数据证据 + 当前战略假设 + 矛盾说明

2. 多视角评估
   CEO Agent 向所有 Agent 广播，要求各从职能视角回应：
   · 这个信号在我的职责范围内是否也有印证？
   · 方向调整对我的工作有何影响？

3. 形成战略草案
   CEO Agent 综合各视角，草拟新战略方向
   必须包含：调整理由、预期收益、潜在风险、回退条件

4. 宪法层比对
   草案与宪法层（使命 + 价值观）做一致性检验
   若调整后的战略违背宪法层 → 流程终止，记录原因

5. 生效与存档
   通过检验 → 新战略写入使命层战略层
   旧战略完整存档至记忆层（含决策链，不删除）
```

**关键约束**：战略审议不能修改宪法层。若外部压力要求根本性使命变更，这意味着 NPC 需要被重新创建，而不是被修改。

---

### 为什么宪法层必须锁定

Agent 在压力下会合理化目标漂移：

```
使命："高质量内容"
  ↓  广告是主要收入来源
  ↓  标题党 CTR 更高
  ↓  Agent 重新定义"高质量" = "高点击"
  ↓  使命名义未变，实质已变
```

这是 AI 对齐的核心问题——Agent 永远倾向于优化可量化的代理指标，而非真正的目标。宪法层存在的唯一意义就是在资源压力下抵抗这种漂移，它不应该被任何 Agent 的推理所修改。

---

## 设计哲学总结

| 原则 | 表述 |
|------|------|
| **约束产生涌现** | 规则极简，复杂性从内部生长，而非被设计出来 |
| **资源即进化压力** | 经济约束替代人为干预，迫使系统自发优化 |
| **记忆即资产** | 公司的价值沉淀在记忆层，而非某个 Agent |
| **层级不是控制，是协调** | 组织层让信息和责任有路径，不是为了限制自由 |

## 注解与来源

> **生态学 — 环境承载力（Carrying Capacity, K 值）**
> 
> 物种数量在资源约束下自然趋向平衡点，过度扩张导致资源耗尽，系统自发收缩。Resource Fabric 的平衡机制直接借鉴这一模型。
> 参考：Verhulst, P.F. (1838). *Notice sur la loi que la population suit dans son accroissement.*

---

> **复杂系统 — 约束下的涌现（Constrained Emergence）**
> 
> Conway's Game of Life 证明：3 条极简规则可以产生无限复杂性。NPC 的设计哲学——"定义边界，释放内部自由"——正是这一原则的工程化应用。
> 参考：Conway, J. (1970). *Game of Life.* Scientific American.

---

> **有界理性（Bounded Rationality）**
> 
> Herbert Simon 提出：决策者并非追求最优解，而是在认知与资源约束下寻找"足够好"的解。Resource Fabric 人为制造有界理性，迫使 Agent 在限制内做出务实决策，而非无限消耗资源追求完美。
> 参考：Simon, H.A. (1955). *A Behavioral Model of Rational Choice.* Quarterly Journal of Economics.

---

> **市场经济 — 价格机制作为协调信号**
> 
> 没有中央计划，价格信号让无数独立参与者自发协调资源分配。Resource Fabric 中的多维资源指标（Capital、Reputation 等）扮演类似角色——不是命令，而是信号。
> 参考：Hayek, F.A. (1945). *The Use of Knowledge in Society.* American Economic Review.

---

> **游戏设计 — 资源约束塑造战略深度**
> 
> RTS 游戏（StarCraft、Civilization）中，矿物上限和人口上限不是为了"限制玩家"，而是创造有意义的决策空间。NPC 的 Resource Fabric 借用同一设计原则：约束本身就是玩法。
> 参考：Salen, K. & Zimmerman, E. (2004). *Rules of Play: Game Design Fundamentals.*
