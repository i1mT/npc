计划文档用于在动手前明确问题、目标、影响面、实现步骤和验证方式，避免边做边扩大范围。

## 命名规范

计划文件按以下格式命名：

```txt
docs/plan/{时间}/{id}.md
```

示例：

```txt
docs/plan/2026-06-20/split-sim-db-module.md
```

## 适用场景

- 涉及多模块、多页面或数据结构变更。
- 涉及 Mastra Agent、工作流、模拟公式、董事会或七层模型的行为变化。
- 涉及文件拆分、架构调整、测试策略或部署方式变化。
- 用户明确要求先计划再实现。

## 计划模板

### 1. 背景

- 要解决什么问题。
- 问题影响哪些页面、角色、Agent、数据表或业务流程。
- 为什么现在需要处理。

### 2. 当前现状

- 当前功能、代码、数据或流程是什么样子。
- 现有方案有哪些限制、风险或维护成本。
- 已知依赖、历史包袱和不能破坏的行为。

### 3. 目标

- 本次迭代完成后应达到什么状态。
- 明确成功标准。
- 明确不在本次处理范围内的内容。

### 4. 实现方案

- 模块拆分：涉及哪些目录、文件、API 或数据表。
- 数据流：事件、快照、变更、DTO、页面之间如何协作。
- UI 流程：页面结构、关键交互、状态展示和设计系统约束。
- 实现步骤：按可提交、可验证的顺序拆分任务。
- 风险控制：兼容、迁移、回滚或降级方式。

### 5. 验证方案

- 测试用例：正常路径、异常路径和边界条件。
- 测试方式：构建、脚本验证、手动验证或未来单元测试。
- 验收标准：怎样判断实现已经满足目标。

### 6. 交付动作

- 代码：变更范围和关键文件。
- 文档：需要同步更新哪些 `AGENTS.md` 或 `docs`。
- 验证：最终运行了哪些命令，结果如何。
- 后续事项：本次不处理但需要跟进的问题。
