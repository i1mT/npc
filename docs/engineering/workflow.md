# 开发流程

开发流程用于保证需求实现可追踪、代码评审有依据、合并质量可控。

## 标准流程

1. 阅读根目录 `AGENTS.md` 和受影响目录的 `AGENTS.md`。
2. 复杂多步任务先创建 todo，持续跟踪进展。
3. 明确目标、范围、影响面和验证方式；大型改动先在 `docs/plan/{date}/{id}.md` 写计划。
4. 按现有分层完成变更，避免把 UI、数据访问、模拟规则和 Agent 编排混在同一层。
5. 同步更新相关产品、工程、设计或上下文文档。
6. 运行必要验证并记录结果。

## 变更要求

- 动态语言代码文件尽量不超过 400 行。
- 每层目录直接文件尽量不超过 8 个，超过时应规划子目录。
- 遇到僵化、冗余、循环依赖、脆弱性、晦涩性、数据泥团或不必要复杂性，应提示用户并给出优化建议。
- 不要随意清理用户未要求的业务变更；若工作区已有未跟踪或未提交内容，应只处理当前任务相关文件。

## 文档联动

- 目录职责变化：更新对应目录的 `AGENTS.md`。
- 核心架构或运行方式变化：更新根目录 `AGENTS.md` 与工程文档。
- 产品体验变化：更新 `docs/product/AGENTS.md`。
- UI 风格、组件规范变化：更新 `docs/design/AGENTS.md` 或 `docs/design/design-system.md`。
